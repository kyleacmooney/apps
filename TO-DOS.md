- **Make the app easy to set up and use with Claude + Supabase MCP** - The intended workflow is that users interact with the app via Claude using the Supabase MCP connector. - **Add per-user Supabase support (bring your own backend)** - Allow users to connect their own Supabase project for full data ownership. ## True Private Chat — Bypass Shared Edge Function - 2026-03-11 02:31

- **Enable fully private AI chat without routing through shared Supabase** - In "private" token storage mode, the user's Claude OAuth token is still sent in the request body to Kyle's shared `chat` edge function. The token transits through infrastructure the user doesn't control. **Problem:** Private mode currently only controls *where the token is persisted* (user's own Supabase `user_secrets`), but the actual API call still goes `browser → shared edge function → Anthropic`. The token is exposed to the shared edge function at `chat/index.ts:91,114,153`. A truly private mode should never send the token to Kyle's infrastructure. **Files:** `supabase/functions/chat/index.ts:110-115` (token resolution — `local`/`private` modes send token in body), `src/pages/Chat.tsx:349-357` (browser sends token + storage mode to shared edge function), `src/lib/token-storage.ts:82-95` (private read from user's Supabase). **Solution:** Options: (1) Let users deploy their own `chat` edge function on their own Supabase project and configure the app to call it instead of the shared one; (2) Call the Anthropic API directly from the browser in private mode (simpler but exposes token to client-side code, which is already the case in `local` mode); (3) Provide a deploy script/guide so users can stand up their own edge function easily. Option 2 is the simplest — skip the edge function proxy entirely when in private/local mode and call `api.anthropic.com` directly from the browser using the user's OAuth token. The edge function mainly exists to keep the token server-side in `shared` mode.
